## 目的

記事を追加したときに、本文表示、OG 画像、内部リンクの不整合をデプロイ前に検出できる状態を作る。

変更頻度の高い記事処理と Markdown 表示に自動テストを置き、依存パッケージの更新や UI 改修による退行を検出できる状態を作る。

## 実装方針

- 既存の見た目、URL、記事本文は、各タスクで変更対象として明記した箇所以外は変えない。
- 単一用途の抽象化を先回りして増やさず、実際に重複している処理か、単体テストが必要な純粋ロジックだけを分離する。
- リファクタリング前後で `typecheck`、`lint`、テスト、ビルドを成功させる。
- 新しい依存パッケージは、標準 API と既存依存だけでは実装が複雑になる場合に限って追加する。
- コンテンツの不備は実行時の既定値で隠さず、対象ファイルと項目名を含むエラーとしてビルド前に報告する。

## P0：記事データの解析と検証を一本化する

### 背景

Web 側の `app/lib/blog.server.ts` と OG 画像生成側の `scripts/generate-og-images.tsx` は、別々の型と frontmatter パーサーを持っている。

Web 側は欠損項目を空文字や `1970-01-01` に置き換えるが、OG 画像生成側は frontmatter 自体がなければ失敗する。

同じ Markdown が実行経路によって異なる記事データになるため、項目追加や検証強化のたびに両方を修正する必要がある。

### 設計

- [ ] 実行環境に依存しない記事モデルと解析処理を、共有モジュールへ移す。
- [ ] 共有モジュールは `import.meta.glob`、Node.js のファイル API、React を参照しない純粋な TypeScript とする。
- [ ] `BlogPost` と `BlogPostSummary` を `.server.ts` から共有モジュールへ移し、クライアントコンポーネントがサーバーモジュールの型を import しない状態にする。
- [ ] Web 側には `import.meta.glob` から文字列を取得する責務だけを残す。
- [ ] OG 画像生成側にはファイル一覧とファイル内容を読み込む責務だけを残す。

### frontmatter の契約

- [ ] `title` は空白だけではない文字列を必須とする。
- [ ] `date` は `YYYY-MM-DD` 形式の文字列を必須とする。
- [ ] `date` は形式だけでなく、実在する暦日であることも検証する。
- [ ] `description` は空白だけではない文字列を必須とする。
- [ ] `tags` は文字列の配列を必須とする。
- [ ] 空文字のタグと同一記事内の重複タグをエラーにする。
- [ ] `draft` は boolean を必須とし、文字列の `"false"` を false として扱わない。
- [ ] slug は Markdown のファイル名から生成し、空文字と重複を許可しない。
- [ ] 公開記事は空の本文を許可しない。
- [ ] 未知の frontmatter 項目をエラーにするか警告にするかを決め、テストで固定する。

- [ ] YAML の構文エラーと項目値の検証エラーを区別する。

### テスト
