この記事も人力で書いてます。AI、どけ、俺がやる。

タイトルの通り、ドキュメント DB で count 取る方法のベストエフォートがいまだに分からない。

昔リリースしていた個人ゲームアプリで、firebase の Realtime Database を採用していた。なぜ採用したのかはもう覚えていない。そのころは「これからはドキュメントDBでしょ！」みたいな流れがたしかにあったんだ。RDB -> NoSQLDB -> GraphQL -> PostgreSQL みたいな流れがたしかにあったんだ。

それで、その個人アプリではユーザが数百万人程度いたんだけど、ドキュメント DB を何も理解せずに使っていた僕は、

「ユーザの総数出したいな？ん？これ全件取得しないと出せないじゃん。まぁええやろ。ほい実装。」

となり、無事に課金大爆死をした。アップデートした日、寝て起きたら課金アラートのメールが大量に来ていて「ん？なぜ？」となった。

# その時の対応方法

その頃はAIエージェントなんかなかったから、冷や汗書きながら一生懸命調べて、count取るのはreadが大量に発生するという事に気が付き、いろんな知見をネットから拾ってきた結果、総数用のドキュメントを別途用意し、そこで総数を運用しろという事を知った。

でもこの方法、なんだか腹落ちしていない。なぜ別途持たなきゃいかんのだとずっと思っていたし、今でも思っている。そういうものだという事と、自分の中で気持ちの良いロジックだと思うことは別なんだ。

# その後 firebase Firestore が出た

こっちはJSONツリー型DBの Realtime Database と違って完全にドキュメント DB だった。こっちはちゃんと count() があった。でも結局最安の方法ではなかった。
Realtime Database に比べるとかなり安くなっているが、結局index scan数による課金なので、ユーザが百万を超えた時点で、些細な課金の差が、赤字に繋がった。

個人アプリで大切なのは、パフォーマンスもとてもとても大切だけど、やはりいかにクラウド課金量を減らせるかが一番なのだ。マネタイズしない趣味アプリでは無い限り、赤字のアプリを運営し続けるのはただのドM野郎なのだ。

# なぜしっくりこないのか

RDB で総数用のテーブルを用意する運用している人いないでしょ？それが全て。

総数なんて、アプリで普通に使いたい値なのに、知らずにドキュメント DB を採用して、「ええ！？総数取るの苦手なの！？」ってなったあの昔の思い出がトラウマになっているのかもしれない。10万ぐらい課金で飛んだのもまじで忘れない。

いまは PostgreSQL でしょみたいな流れの世の中なので、わざわざドキュメント DB を採用する人は少ないかもしれないけど、もしもドキュメント DB を採用してみたいなと思っている人は是非手元の AI にドキュメント DB の得意な事苦手な事を聞いてから採用してほしい。

# 今は？

今は Cloudeflare に移行したので、普通に D1 データベースで select 打ってます。

結局は、こちらも read は rows read ベースの課金なので、count() がどれぐらいの行数読むか次第ですが、ドキュメント DB より課金が優しいし、実行計画とかインデックス等である程度チューニングできるので、そこまで不満はないです。

# おわりに
