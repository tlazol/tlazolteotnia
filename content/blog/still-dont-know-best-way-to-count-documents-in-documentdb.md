この記事も人力で書いている。AI、どけ、俺がやる。

タイトルの通り、ドキュメント DB で count を取るベストな方法がいまだに分からない。

昔リリースしていた個人ゲームアプリで、Firebase の Realtime Database を採用していた。なぜ採用したのかはもう覚えていない。その頃は「これからはドキュメント DB でしょ！」みたいな流れがたしかにあったんだ。データベースは RDB -> NoSQL DB -> PostgreSQL、API は GraphQL、みたいな流れがたしかにあったんだ。

それで、その個人アプリではユーザが数百万人程度いたんだけど、ドキュメント DB を何も理解せずに使っていた僕は、

「ユーザの総数を出したいな？ ん？ これ全件取得しないと出せないじゃん。まあええやろ。ほい実装。」

となり、無事に課金大爆死をした。アップデートした日、寝て起きたら課金アラートのメールが大量に来ていて「ん？なぜ？」となった。

## その時の対応方法

その頃は AI エージェントなんかなかったから、冷や汗をかきながら一生懸命調べた。Realtime Database は read 回数ではなく、保存量とダウンロード量で課金される。僕の実装は count を取るために全件をダウンロードしていたのだ。いろんな知見をネットから拾ってきた結果、総数用のノードを別途用意し、そこで総数を運用しろということを知った。

でもこの方法、なんだか腹落ちしていない。なぜ別途持たなきゃいかんのだとずっと思っていたし、今でも思っている。そういうものだということと、自分の中で気持ちの良いロジックだと思うことは別なんだ。

## その後 Cloud Firestore が出た

こっちは JSON ツリー型 DB の Realtime Database と違って、完全にドキュメント DB だった。リリース後しばらくして count() が実装されたので、おお！となったのを覚えている。

でも結局、最安の方法ではなかった。

Realtime Database で全件を取得する方法に比べるとかなり安くなっていたが、count() は読み取った index entry 1,000 件ごとに 1 read として課金される。ユーザ 100 万人分のドキュメントを一度数えると 1,000 read だ。これを頻繁に呼ぶ僕の使い方では、そのわずかな差が赤字につながった。

結局は総数用のテーブルを用意した。

個人アプリでは、パフォーマンスもとてもとても大切だけど、やはりいかにクラウドの利用料金を減らせるかが一番大切なのだ。マネタイズしない趣味アプリでもない限り、赤字のアプリを運営し続けるのはただのド M 野郎なのだ。

## なぜしっくりこないのか

総数なんて、アプリで普通に使いたい値なのに、知らずにドキュメント DB を採用して、「ええ！？ 総数取るの苦手なの！？」ってなったあの昔の思い出がトラウマになっているのかもしれない。10 万円ぐらい課金で飛んだのもまじで忘れない。

今は PostgreSQL でしょ、みたいな流れの世の中なので、わざわざドキュメント DB を採用する人は少ないかもしれないけど、もしもドキュメント DB を採用してみたいと思っている人は、ぜひ手元の AI にドキュメント DB の得意なことと苦手なことを聞いてから採用してほしい。
