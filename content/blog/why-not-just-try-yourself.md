# 自分が学習する時によく思うこと

この記事も自分で書いている。
生成 AI に文章を書かせることに慣れると、自分では書けなくなる。
毎週、日記の宿題に取り組む息子にも負けてしまう。

技術やアーキテクチャを理解するには、やはり自分で試してみなければならないという話。

参考書で学び、資格試験に合格したものの、実務経験のない人を否定したいわけではない。
自分の学習方法の根底に「まずは触ってみないと何もわからない」があるという話だ。

自分も資格の勉強をするときに参考書は読むのだが、「本を読むだけではわからない」と思ってしまい、そこから進めなくなる。
まずは触ってみたいのだ。
何もわからない状態でもよいので、手探りで満足するまで触ったあとに、参考書を読んで知識を固めたい派なのだ。

## 絵文字リアクションの再現で見えた論点

Slack や Discord など、最近のコミュニケーションツールには、たいてい絵文字リアクションがある。
今では、とてもなじみ深い UI だと思う。

しかし、「どのようなデータ構造にして、どのようなUI/UXにすれば同じものを作れるのか」と問われると、すぐには答えられない。

自分で再現しようとしたときに初めて、ぱっと見ではシンプルな機能であっても、多くのことを考慮しなければならないと気づけるのだ。

最初に思いついたのは、次のような点だった。

- データベースの構造
- UI 操作の連打と重複への対応
- 絵文字リアクションがアプリケーション全体のパフォーマンスに与える影響
- クラウドサービスの利用料金
- 絵文字リアクションを促すための UI 表現

実際に実装してみると、さらに多くの気づきと学びが出てくる。
これこそが、知識が血肉となる瞬間なのだ。
個人的には、この経験がなければ、参考書でいくら知識を得てもなかなか身につかないのだ。

## MVP として実装した範囲

今回は、MVP として、絵文字リアクションを追加できるところまで実装した。
リアクションの取り消しは未実装である。

UI 操作の連打防止や、大量の絵文字を DOM に表示しても描画性能が落ちにくい仕組みなど、いくつかの点を考慮した。
その中でも最も時間をかけたのは、Cloudflare D1 の利用量を無料枠に収めるための設計だ。

## D1 のテーブル構成

絵文字リアクションでは、投票履歴と集計値を別々のテーブルに保存している。

| テーブル        | 役割                                                                       |
| --------------- | -------------------------------------------------------------------------- |
| reaction_votes  | 訪問者ごとの投票履歴を保存し、複合主キーによって同じ絵文字の重複投票を防ぐ |
| reaction_counts | 記事と絵文字ごとの投票数を保存し、表示時の再集計を不要にする               |

`reaction_votes` に投票が追加されると、`increment_reaction_count` トリガーが `reaction_counts` の集計値を更新する。

## D1 の読み取り行数を抑える設計

このリアクション機能では、D1 に問い合わせる回数だけでなく、クエリが読み取る行数も減らしている。

D1 の利用量は、返却行数ではなく、クエリが読み取った行数で計測される。
そのため、クエリの実行回数を減らすだけでは十分ではない。

### 投票履歴を毎回集計しない

表示のたびに次のような集計を実行すると、投票数が増えるほど読み取る行数も増える。

```sql
SELECT emoji, COUNT(*)
FROM reaction_votes
WHERE post_slug = ?
GROUP BY emoji;
```

この実装では、`INSERT` トリガーが集計済みの `reaction_counts` を更新する。

表示時に読み取るのは大量の投票履歴ではなく、記事ごとに最大 16 種類しかない絵文字別の集計値だ。
投票数が 100 件から 10 万件へ増えても、表示クエリが返す行数は絵文字の種類数を超えない。
この事前集計によって、D1 の読み取り行数を抑えている。

### 複合主キーを検索条件に合わせる

`reaction_counts` の主キーは、`(post_slug, emoji)` である。
集計値の取得では先頭列の `post_slug` を検索条件にするため、全件走査を避けて対象記事の行だけを探せる。

```sql
SELECT emoji, count
FROM reaction_counts
WHERE post_slug = ?;
```

`reaction_votes` の主キーは、`(visitor_hash, post_slug, emoji)` である。
訪問者の投票状態を確認するときは主キーの先頭 2 列を指定するため、全訪問者の投票履歴を読む必要がない。

```sql
SELECT emoji
FROM reaction_votes
WHERE visitor_hash = ?
  AND post_slug = ?;
```

### 記事一覧に必要なデータを一括取得する

トップページでは、記事ごとに D1 へ問い合わせない。
全記事のスラッグを一つの `IN (...)` クエリにまとめ、集計値と訪問者の投票状態をそれぞれ一括取得する。

```sql
SELECT post_slug, emoji, count
FROM reaction_counts
WHERE post_slug IN (?, ?, ...);
```

これにより、記事数を `N` としたときのクエリ数を `2N` 本から 2 本へ固定している。

### `batch()` で通信の往復をまとめる

集計値と訪問者自身の投票状態を取得する 2 本の SQL 文は、`db.batch()` を使って D1 へ一度に送る。

```ts
const [countsResult, votesResult] = await db.batch([countsQuery, votesQuery]);
```

`batch()` 自体は D1 の読み取り行数を減らさない。
2 本の SQL 文を 1 回の呼び出しにまとめることで、Cloudflare Workers と D1 の間の通信回数と待ち時間を減らしている。

### 必要な列だけを取得する

クエリでは `SELECT *` を使わず、画面の生成に必要な列だけを取得している。
これは D1 の読み取り行数を減らす仕組みではないが、D1 から Cloudflare Workers へ返すレスポンスのサイズとシリアライズ処理を抑えられる。

この実装では、`INSERT` トリガーによる事前集計、検索条件に合わせた複合主キー、記事一覧の一括取得を組み合わせた。
これにより、投票数や記事数が増えても、D1 が読み取る行数とデータベースへの通信回数が増えにくくなる。
